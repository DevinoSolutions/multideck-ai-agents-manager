# magent — agent guide

Open every project in its own terminal, launch an AI agent in it, and auto-tile every window across every monitor. Entry point: `magent = magent.cli:main` (pyproject.toml). User-facing docs live in README.md and `magent docs` (generated config-schema reference); architectural rationale and the known-debt ledger live in DESIGN.md — this file is the terse map for a cold agent session, not a restatement of either.

## Commands

- Setup: `pip install -e ".[dev]"` (uv equivalent: `uv sync --extra dev`; `uv.lock` is committed). One-time: `npm install` to activate the husky pre-commit hook. Sentry error reporting ships in the base install (sentry-sdk is a base dep; the `[sentry]` extra survives only as a compat alias) and stays env-gated via `MAGENT_SENTRY_DSN`.
- Quality gate — **must pass before every commit**: `uv run python scripts/check.py` (full, default) or `uv run python scripts/check.py --fast` (pre-commit hook, ~5 seconds). **Full** runs: ruff check + ruff format --check + custom lint (MD001-MD005) + ty strict (src+scripts, plus a second `--python-platform win32` pass scoped to `platform/windows.py`+`hotkey.py`) + compileall + vulture + pytest unit w/ cov. **Fast** runs: ruff + format + custom lint + both ty passes (skips compileall/vulture/pytest). The husky pre-commit hook runs `--fast` automatically; the husky pre-push hook runs the full gate (no `--fast`) as a second line of defense. Pytest enforces a `--cov-fail-under=52` coverage floor (calibrated to the "Lint + Type Check + Coverage" CI job's ubuntu-latest run, not a local Windows box — `platform/windows.py`/`hotkey.py` are 0%-coverable there, pulling total coverage down several points vs. Windows) and `--strict-markers` (an unregistered `@pytest.mark.*` is a hard error, not a silent no-op). CI's `quality` job runs `uv lock --check` then the full gate via `uv sync --extra dev --frozen --python 3.13`; the separate `unit` job runs the OS x Python matrix (3.10–3.14, windows/macos/linux) via plain `pip install -e ".[dev]"`. Two advisory security jobs ride alongside (not in the local gate, by design — see DESIGN.md): a pinned-CLI gitleaks secret scan on every push/PR, and `dependency-audit.yml` running pinned `pip-audit` on dependency changes + a weekly schedule; `.github/dependabot.yml` files weekly update PRs (uv / github-actions / npm).
- Unit tests: `pytest tests/unit/ -q` — last measured 569 passed, 2 skipped on this branch; re-run rather than trusting that count indefinitely.
- Single test: `pytest tests/unit/test_x.py::test_y`.
- e2e locally: `pytest tests/e2e/ -m "e2e and not needs_ssh"`. WARN: a bare `pytest` also collects `tests/platform/` (needs a real display/terminal), the `needs_ssh` e2e tests, and `tests/dist/` (builds a real wheel each session) — CI splits these into separate jobs/legs; don't run bare `pytest` locally and read a clean exit as "the gate passed".
- The `needs_ssh` tier is CI-only by design: `tests/e2e/test_ssh.py` (dry-run pins) + `tests/e2e/test_ssh_real.py` (REAL wire — every test traverses the loopback sshd that `.github/actions/setup-ssh-server` provisions, exporting `MDTEST_SSH_PORT`/`MDTEST_SSH_KEY`/`MDTEST_SSH_HOST` and an `mdssh` alias in the runner's real `~/.ssh/config`). Locally these skip because the env vars are absent — **never install/enable an SSH server on a dev machine to run them**. Tests that seed the ssh user's real HOME (the Windows attach flagship — attach can't inject `--config` remotely) are additionally gated on `GITHUB_ACTIONS=true` / `MDTEST_ALLOW_REAL_HOME=1`. macOS window-over-SSH legs are a loud `::warning` skip (see DESIGN.md known debt).
- dist (packaged-install) locally: `pip install build` (not a dev extra; CI installs it per-job) then `pytest tests/dist/ -m dist` — builds a wheel, installs it into a pristine no-extras venv, and drives the real installed `magent` entry point (proves the exact `pip install` user env; import-sweep catches dev/optional-dep leaks). Window-spawn leg is win32-only and skips elsewhere; wheel build + import sweep + virgin first-run + socket-real serve run on every OS.
- Real-PTY menu tier (`tests/e2e/test_pty_menu.py`, marker `pty`, rides the existing `end-to-end` CI job on all 3 OSes): drives the installed `python -m magent` under a genuine pseudo-terminal — pexpect on POSIX, pywinpty/ConPTY on Windows (`pywinpty>=2.0; sys_platform=='win32'` in the `dev` extra) — via the uniform driver `tests/e2e/_pty.py`, asserting on the plain on-screen text (ANSI stripped; children run `NO_COLOR=1`) and, for first-run, the valid config written to disk. This is the only test of the real `sys.stdin.isatty()` interactive path (`CliRunner` fakes the tty). Locally it runs anywhere pexpect/pywinpty resolve.
- Browser-upload tier (`tests/e2e/test_upload_browser.py`, marker `browser`, dedicated `browser-upload` ubuntu job, CI-only, gated on `MDTEST_BROWSER=1`): a real headless Chromium (Playwright — the `browser` extra `["playwright>=1.40"]`, NOT in `dev`, provisioned only by that job via `playwright install --with-deps chromium`) drives the real mobile upload page against a real `magent serve` on loopback and asserts the on-disk upload is byte-identical + the page title/form contract + the Ctrl+V clipboard-paste confirm flow (synthetic ClipboardEvent → staged preview with Send gated on project selection → XHR upload → byte-identical `paste-*.png` on disk). Locally the module skips cleanly (no `MDTEST_BROWSER`, no Playwright). Honest gap: it symlinks real `tmux` as `psmux` and stands up a real tmux session as the multiplexer shim (Linux has no `psmux` binary) — see DESIGN.md.
- The `needs_tailscale` tier is CI-only by design (`tests/e2e/test_tailnet_real.py` + the non-required `tailnet` ubuntu job): it joins a REAL ephemeral, tag-scoped Tailscale node (`tailscale/github-action`, SHA-pinned, OAuth via `TS_OAUTH_CLIENT_ID`/`TS_OAUTH_SECRET` + `tag:ci`) and exercises `tailnet.py`'s live `tailscale` probes (`ip4`/`magicdns_host`/`probe`) plus `magent serve`'s Tailscale-facing default bind (`/health` answers on the real Tailscale IP; the LAN wildcard is proven unbound). Gated on `MDTEST_TAILSCALE=1` — locally it skips because the var is absent; **never install/join Tailscale on a dev machine to run it** (same posture as `needs_ssh`). The OAuth secrets don't exist yet: the job detects their absence and skips LOUDLY (`::warning`, job stays green, forks never fail) — once the user adds them it goes fully live with zero code change (one-time setup steps in the job comment). Excluded from the `end-to-end` job's SSH-less run exactly like `needs_ssh`.
- The `monitor_lab` tier is CI-only by design (`tests/platform/test_monitor_lab_tiling.py` + the `monitor_lab` CI job, windows-latest): it INSTALLS the parsec-vdd virtual-display driver to fabricate a mixed-DPI multi-monitor topology, then drives real `--go` tiling across the virtual monitors and asserts each window rect lands in its `compute_grid` cell (physical pixels). Gated on `MDTEST_MONITOR_LAB=1` — locally it skips because the var is absent; **never install the driver / set the var on a dev machine to run it** (same posture as `needs_ssh`). The lab engine is `tests/platform/monitor_lab.py` (stdlib ctypes, win32-only, import-harmless on POSIX). The offline half — `tests/unit/test_monitor_lab_topologies.py` feeding committed golden topologies (`tests/platform/fixtures/topologies/*.json`) into `compute_grid` — runs everywhere in the normal unit gate. Two more CI-only Windows interaction tiers ride in the platform-integration windows leg (same never-on-a-dev-box posture): the `wt_cold` tier (`tests/platform/test_wt_cold_start.py`, gated `MDTEST_WT_COLD=1`) pins the first-ever `wt` launch on the fresh runner against the default settle+retry tiling budget — coldness is guaranteed by CI step ordering, so it must stay the first step that can spawn wt; and the `interaction` tier (`tests/platform/test_real_hotkey.py`, gated `MDTEST_INTERACTION=1`) drives the REAL Alt+V chain — product-spawned listener, real `SendInput` chord, real CF_DIB clipboard, real `serve` + psmux inject — and would stomp a dev machine's clipboard/keystrokes, so never set the var locally. The windows leg also installs pinned psmux (v3.3.6 release zip, sha256-verified, into `%LOCALAPPDATA%\psmux`) so `tests/platform/test_real_psmux.py` (lifecycle + full attach chain in a real wt window) runs instead of skipping. Riding in the same CI job (shared `lab` fixture + driver install via `tests/platform/conftest.py`): the doctor-replay tier — `magent doctor --json` emits monitor topology under a `monitors` key, `tests/platform/doctor_replay.py` plans an achievable lab topology from a bug-report blob, `tests/platform/test_doctor_replay.py` replays committed sample reports (`tests/platform/fixtures/doctor_reports/*.json`) live, and `tests/unit/test_doctor_replay_offline.py` pins the parse/plan/slot math (incl. negative origins) everywhere.
- Real-writer contract tier (`tests/e2e/test_state_sink_contract.py`, marker `node_contract`, rides the existing `end-to-end` job on all 3 OSes — node is preinstalled on every hosted runner, so no CI change): `npm install`s the pinned real agent-state writer package (`ai-agent-notifier@1.0.6` — see `AINS_VERSION`) into a tmp prefix, drives its REAL hook (`src/notify.mjs`) under real node with realistic Claude `Stop`/`Notification`/`SessionStart` events and a fully-redirected HOME, then checks magent's real `agent_state` reader against it. It PINS a headline finding (F-NC-001): the package writes ZERO `~/.magent/state/` records (it is a pure notifier; there is no `state-sink.mjs`) — so the tier currently asserts the store stays empty and flips RED if a wired hook ever starts writing records. Skips cleanly if node/npm or the npm registry are unavailable; tmp-dir only, never touches real `~/.magent`/`~/.claude`.
- Nightly soak tier (`tests/e2e/test_soak.py`, marker `soak`, CI-only, gated on `MDTEST_SOAK=1`; dedicated `.github/workflows/soak.yml` — `schedule` nightly + `workflow_dispatch`, ubuntu-latest + windows-latest, non-required): drives a REAL `serve --host 127.0.0.1` + REAL detached `attention -d --interval 1` under a sustained churn loop for `MDTEST_SOAK_SECONDS` (default 1500s ≈ 25 min active, ≤45-min job budget) — cycling the full agent-state vocabulary + periodic real POST /upload & /health round-trips — asserting throughout AND at the end: heartbeat mtime keeps advancing (never stale per `log.HEARTBEAT_MAX_AGE`), uploads accepted byte-identical at minute 25 as minute 1, both pids survive, RSS never runs away (stdlib sampling: `/proc` on Linux, ctypes `GetProcessMemoryInfo` on Windows), log rotation stays bounded, TTL sweep spares fresh records. Locally it skips (var absent); **never set `MDTEST_SOAK` on a dev box** — it deliberately runs tens of minutes. Copies `test_daemon_lifecycle`'s HOME-redirect isolation; a no-op `psmux` shim on the child PATH stands in for a real multiplexer so the upload-accept path is exercised (same device as the browser tier's tmux symlink).
- Run: `magent` (interactive menu), `--go` (skip menu, launch+tile), `--dry-run`, `magent config migrate` (stamps `SCHEMA_VERSION` and persists backfilled project colors — `load_config` itself never writes to disk), `magent serve --host <addr>` escape hatch (default bind is loopback + Tailscale IP, never the LAN wildcard; pass `--host 0.0.0.0` to explicitly restore a LAN-wide bind). Attention stack: `magent attention -d` (daemon: title badges + taskbar flash + optional toast/ntfy per `settings.attention` + `MAGENT_NTFY_TOPIC`; heartbeat shows in `status`, stopped by `down --all`) and `magent watch` (live most-urgent-first table; digit key focuses that window). Both read `agent_state` through `attention.AttentionEngine` — states come from Claude Code hooks / Codex notify.
- `magent status` exit codes (`cli/status.py::status_cmd`): 0 healthy, 1 config missing/invalid, 3 degraded (upload server dead, or listener heartbeat stale). Click itself exits 2 on usage errors — framework behavior, not this codebase's.
- Logs: rotating files under `~/.magent/logs/<name>.log` via `log.get_logger(name)`; names in use: `"launch"`, `"upload"`, `"hotkey"`, `"platform"`, `"attention"`, `"sentry"` (the last only when a DSN is set but sentry-sdk is missing — a broken install, since the SDK is a base dep).

## Architecture

- `cli/__init__.py` is a ~24-line registration hub: imports every command module (running their `@main.command` decorators) and re-exports the names tests/external code reach via `magent.cli.<name>`. `cli/app.py` holds the `main` click group alone, with no top-level imports of sibling command modules, so every command module can `from magent.cli.app import main` without a cycle; the no-subcommand interactive path re-imports its handlers in-body as a documented cycle-break.
- `launch.py::run_magent` is a phase pipeline: `_prepare_grid` → `_select_projects` → `_launch_projects` (per-project dispatch to `_dispatch_ide_project`/`_dispatch_cli_agent_project`) → `_start_psmux_and_upload` → `_tile_targets` (delegates to `tiling.place_windows`). The shell owns the exit code; phases return data.
- `platform/__init__.py`: `Platform` ABC + `get_platform()`. `supports_psmux()`/`supports_hotkey()` default False; `attach_psmux`/`launch_psmux_session` default to `NotImplementedError`. Only `platform/windows.py::WindowsPlatform` overrides those; `macos.py`/`linux.py` implement the abstract methods and take the ABC defaults for the rest.
- `sessions/__init__.py`: the `AGENT_TOOLS` registry (`AgentTool` dataclass: `session_ids`/`resume_command`/`happy`), one `sessions/<tool>.py` per deeply-integrated CLI agent (today: `claude.py`, `codex.py`); also the IDE mirror `IDE_COMMANDS`/`IDE_TOOLS` + `is_ide_tool()`/`ide_command()` — no hand-written `('code','vscode','cursor')` tuples anywhere in `src/`.
- `config.py`: `SCHEMA_VERSION`, `DEFAULT_TOOLS`, the typed dataclasses, pure `load_config` (validates + warns, never writes), `default_config`/`settings_to_dict` (the one envelope factory every generator shares), and `migrate_config_file` (the only function in this module that writes to disk). A project's `windows` field is `list[WindowConfig] | None` (SCHEMA_VERSION 3) — per-window `name`/`tool`/`command` overrides; the legacy `int` / `list[str]` forms still parse and are normalized by `migrate_config_file`. `cli/config_editor.py` hosts the interactive editor (`_config_menu`) and every `config <subcommand>`, including `migrate`.
- `env.py`: the **only module that touches process env** (enforced by TID251 os.environ/os.getenv ban). `MagentEnv(BaseSettings)`: validated MAGENT_* env vars (closed schema, `extra="forbid"`). Dotenv is read from `env.ENV_FILE` (`~/.magent/.env`) **only — never the CWD's `.env`** (magent launches from arbitrary project dirs; their `.env` files must not fail startup). `validation_error_items` is the one env-error formatter (app.py + doctor share it; extra keys report under their raw names, no fabricated MAGENT_ prefix). Host-env accessors (`config_base`, `vscode_storage_base`, `editor_command`, `appdata_dir`, `xdg_config_home`, `localappdata_dir`) for OS-convention variables. `.env.example` drift-pinned by `test_env_schema.py`.
- `sentry.py`: env-gated Sentry wiring (`MAGENT_SENTRY_DSN`, default off); sentry-sdk is a base dependency so a set DSN always works on a healthy install. Errors-only, no PII, logging integration at ERROR level, atexit flush. Zero imports when DSN unset (the SDK loads lazily inside `init_sentry`). DSN set but sentry-sdk missing = broken install; still quiet degradation: a rotating-log warning (never a per-command stderr nag) + a `sentry` check in `magent doctor` carrying the repair hint `pip install --force-reinstall magent-multi-ai-agents-manager`.
- Leaves with no dependency on the cli package: `paths.py` (`find_config`), `style.py` (`style = click.style`), `titles.py` (`MAGENT_TITLE_PREFIX`), `tiling.py` (`place_windows`, shared by launch-path and attach-path tiling), `log.py` (`get_logger` + heartbeat helpers), `grid.py` (`compute_grid`), `tailnet.py` (`ip4`/`magicdns_host`/`probe` — the only module that shells out to `tailscale`), `procs.py` (`pid_alive` — the one pid-liveness check), `psmux.py` (single owner of every psmux subprocess interaction; imports only `magent.log` at module level), `icons.py` (icon/PNG rendering — pure stdlib), `lockfile.py` (the exclusive-lock guard).
- `upload_server.py` imports `launch`, `log`, `paths`, `platform` at top level; the real invariant (LS-A-001) is narrower than "depends on paths only" — it never imports the **cli package**, because `cli/__init__` imports every command module for registration and a reverse import would cycle.
- `hotkey.py` raises `ImportError` at import time off win32 (by design); every caller imports it lazily behind a `supports_hotkey()` check.

## Conventions in force

- Adding a new deeply-integrated agent tool = one new `sessions/<tool>.py` + one `AGENT_TOOLS` entry, with no dispatcher change required — proven by `tests/unit/test_tool_registry.py::TestOneEditExtensionProof::test_adding_a_tool_is_one_dict_entry`. A plain command tool (no session resume) is just a `config.DEFAULT_TOOLS` entry.
- `magent.config.example.json` is drift-pinned to the config factory (`tests/unit/test_config_factory.py::TestExampleConfigMatchesFactory`): any change to `DEFAULT_TOOLS` or a `Settings` default must update the example file in the same commit, or the gate goes red.
- Adding a new platform capability = a default on the `Platform` ABC (a False-returning probe, or a `NotImplementedError` body) plus per-OS overrides; gate call sites with `supports_*()`, never a bare `sys.platform` check in business logic. (The handful of remaining `sys.platform` branches — e.g. `cli/ui.py::_open_in_editor`'s editor-command selection — are genuinely OS-behavioral, not capability gates, and are correctly left alone.)
- Subsystem modules (`launch.py`, etc.) return ints or raise; `sys.exit()` decisions live only in the `cli/` command shells.
- Import policy inside `cli/` command modules: stdlib + leaf imports at top; heavy subsystems (`launch`, `upload_server`, `discover`, `agent_state`, and the platform backends via `get_platform()`) are imported in-body with a one-line why-comment — grep `heavy subsystem: in-body per policy` to find every site. Reason: the registration hub imports every command module eagerly, so a top-level heavy import would make `magent --help` pay that subsystem's startup cost. (`hotkey` imports must stay in-body for a second reason: it raises `ImportError` off-Windows.)
- Tests: JSON-body assertions read `result.stdout`, never `result.output` — Click 8.4's `CliRunner` merges stdout+stderr into `.output`, so a stderr diagnostic (e.g. the config version warning) can silently corrupt `json.loads(result.output)` (NF-S3-002). Characterization/pin tests are written and green BEFORE the refactor they pin, not after. Shared fixtures `runner`/`fake_platform` and the `FakePlatform` test double live in `tests/conftest.py`; CLI tests pass `--config <tmp_path>` rather than monkeypatching the config location.
- `style(...)` comes from `magent.style`; there is no local `S = click.style` anywhere in the repo — the historical alias was renamed out repo-wide. Enforced by TID251 `click.style` ban.
- `typing.Any` is banned repo-wide (TID251 + ANN401). Use `object` + `isinstance` narrowing. Zero `Any` annotations in `src/`.
- `os.environ` / `os.getenv` are banned outside `env.py` (TID251). All env access goes through `magent.env` accessors.
- Window titles follow the grammar in `titles.py` — `magent:` prefix + optional `[!]`/`[x]`/`[+]` state badge + name. Producers call `titles.make_title` (launch-path terminals, psmux attach, `cli/attach.py`); consumers call `titles.parse_title` (`hotkey.py::project_from_title`, tiling's `magent-name` placement mode, which still accepts the pre-rename `md-name` alias). Never hand-build or hand-parse a `magent:` string; there are no `f"magent:` literals anywhere in `src/`. psmux *session names* stay unprefixed — the grammar applies only at the window-title boundary.
- Raw-dict config I/O (round-trips unknown/unmodeled keys) lives only in `cli/config_io.py` (`_load_raw_config`/`_save_raw_config`, used by the interactive editor). Every CLI call site loads typed config through `config_io._load_config_or_exit`; its `as_json=True` mode emits a `{"ok": false, "error": ...}` envelope on stdout — adopted by both `status --json` and `up --json` — instead of the plain-text stderr `Error:` line (NF-S3-005; the former `up_cmd` raw-loader exception is gone).
- `agent_state.py` record schema (`{state, ts, cwd, session_id}`) is pinned by `tests/unit/test_agent_state.py::TestSchemaContract` — any change to `write_state`'s payload shape breaks that test before it can break the out-of-repo writers (`state-sink.mjs`, Codex notify). On a deliberate schema change: bump `RECORD_VERSION`, update the contract test, and update external writers in lockstep. HEADLINE CAVEAT (F-NC-001, see DESIGN.md ledger): the named `state-sink.mjs` writer does **not** exist — `tests/e2e/test_state_sink_contract.py` (marker `node_contract`) drives the real published `ai-agent-notifier@1.0.6` under node and proves it writes ZERO `~/.magent/state/` records (it is a pure notifier). So `TestSchemaContract` currently pins only what magent itself writes/reads; no live external writer is validated against it. The `node_contract` tier flips RED the day any wired hook starts writing state records — that is the signal to promote it into a real positive schema-contract test.
- Commit style is conventional (`feat(scope): ...`, `fix`, `refactor`, `test`, `chore`, `ci`, `docs`) — check `git log --oneline` for the pattern in force.

## Gotchas

- This machine's shell hook (`rtk`) filters shell output; if a result looks surprising, re-run via `rtk proxy <cmd>` for the raw output — it has been observed to mislabel a pytest `1 xfailed` as `1 failed`.
- A file literally named `nul` reappears at the repo root from `> nul` redirects under Git Bash on Windows. Delete it; never commit it (same for the untracked `vscode_debug.txt`).
- Click 8.3 vs 8.4 render a bare `--help` usage line differently (`[COMMAND]` bracketed vs not, for `invoke_without_command=True` groups). `tests/unit/test_cli_structure.py::_normalize_help` normalizes exactly that one substring so the help-snapshot pins pass under either installed version.
- ty is pinned to `ty==0.0.59` (pre-1.0; Dependabot moves the pin, CI's full gate proves each bump). It has known false positives for `isinstance(x, dict)` narrowing → `dict[Unknown, Unknown]` instead of `dict[str, object]`. These are suppressed with per-line `# ty: ignore[invalid-return-type]` comments; 0.0.59 still carries the gap. Canary for "is it fixed yet": the gate runs `ty check … --error-on-warning`, so a version that resolved the narrowing would make the now-unused ignores fail the gate — a green gate on a bump means the suppressions are still load-bearing. When ty 1.0 ships and resolves these, remove the suppressions.
- vulture's `ignore_names` whitelist (pyproject.toml) documents 4 framework-required parameters; add new entries only with a trailing reason comment. Dead code = delete it, don't whitelist it.
- The runtime dotenv lives at `~/.magent/.env` (`env.ENV_FILE`) — a repo-root `.env` is inert (and still gitignored if one appears). `.env.example` is committed and drift-pinned to `MagentEnv` fields by `test_env_schema.py`. A new env var without an `.env.example` update = red gate.
- A config file with no `version` field (or an old one) prints a stderr warning on every load until `magent config migrate` is run; `load_config` itself never writes.
- `qrcode` (used by `cli/ui.py::_print_qr` for `magent mobile`) is an optional dependency: imported in a try/except with an install tip. This is deliberate — don't "declare" it as a missing dependency.
- `ruff format` was applied repo-wide as one isolated `style:` commit; its SHA is recorded in `.git-blame-ignore-revs`. Run `git config blame.ignoreRevsFile .git-blame-ignore-revs` once so local `git blame` skips it (GitHub honors the file automatically). `ruff format --check` is a gate step, so formatting stays enforced.

## Hard boundaries

- Never `git add` the local `nul` file, `vscode_debug.txt`, the `audit/` directory, `graphify-out/`, or `magent.config.json` (personal, gitignored — `magent.config.example.json` is the committed sample).
- `scripts/check.py` must exit 0 before every commit.
- The upload server has no auth token by design — the loopback + Tailscale bind IS the access control. Do not add a token; do not widen the default bind.
- Tracked, deliberate debt lives in DESIGN.md's known-debt ledger — do not drive-by fix ledger items. (The four `NF-S3` debts once fenced here — `cli/status.py::_menu_down`'s unconditional "Stopped upload server." echo, `cli/docs.py::_generate_docs`'s fabricated `"aider"` tool, `cli/attach.py::_attach_nomux`'s `"claude --continue"` literal fallback, and `status --json`'s plain-text config-error shape — were resolved in pass-2, PR-C.)

See README.md for user-facing usage, `magent docs` for the generated config-schema reference, and DESIGN.md for architectural rationale, deliberate-design decisions, and the full known-debt ledger. The PyPI release runbook (tag-triggered trusted-publishing pipeline in `.github/workflows/release.yml`, plus the one-time PyPI/GitHub setup) lives in RELEASING.md.
